# AI 직원 만들기 — 설계 노트

> 비개발자 셀러가 코딩 없이 **대화로** AI 직원/자동화를 만드는 제품. **빌더(`ai-agent-maker`) + 기법 라이브러리** 구조이며, 강의 덱 "AI와 일하는 법"과 함께 자기 PC에서 만들고 쓴다.

이 문서는 **현재 확정된 설계와 그 근거**를 한 장으로 보는 노트다. 코드 변경 전엔 원본(SoT)인 `plugins/ai-employee-coach/skills/ai-agent-maker/SKILL.md`와 `AGENTS.md`를 먼저 본다.

---

## 1. 제품 정체성

- **대상**: 비개발자 셀러. 코딩 없이 **대화로** AI 직원/자동화를 만든다.
- **주 환경**: **Codex**(네이티브 스킬·플러그인, 내장 `image_gen`). 진입점은 루트 `AGENTS.md` + 플러그인 스킬.
- **배포**: GitHub 마켓플레이스 — `codex plugin marketplace add rubydatalab/ai-project-coach` → `codex plugin add ai-employee-coach`.
- **두 부분**: (a) 스킬 패밀리 11종(= 플러그인 `ai-employee-coach`), (b) 강의 덱 `lecture/decks/ai-il/`.
- **수강생에게 *가르치는* 범위 밖**: git·배포·온라인 호스팅·프론트엔드. ※ 이건 **수강생이 다루는 범위** 얘기다 — 이 레포 자체는 GitHub로 배포한다.

---

## 2. 핵심 사상

- 셀러는 **"목표/불편/희망"만** 말한다. "어떻게"는 AI가 분석·제안한다.
  - 보통 직원 = 내가 방법을 고민해 지시 / **AI 직원 = AI가 해결법을 제시**.
- 진행 패턴: **"~하고 싶어"** → AI가 **한 번에 하나씩** 질문 → **추천(최대 3개, 추천 먼저)** → **작게 테스트** → **위험하면 멈춤**.
- **전문용어 금지** — 번역표(4번)로 쉬운 말 통일.

---

## 3. 구조: 빌더 + 기법 라이브러리

- 마스터 `ai-agent-maker`가 대화로 직원을 *짓고*, 필요한 기법(이미지 생성·상세페이지 구성 등)을 끼운다.
- 전문 코치 10종이 특정 작업을 담당. 합 **11종**(마스터 1 + 전문 10).
- **원칙**: do-er(완성품을 쥐어주기)가 아니라 **빌더(만드는 법)**. 비주얼이 필요한 직원은 빌더가 이미지 생성 기법을 끼워 짓는다.
- 전문 코치: `setup-helper` · `automation-finder` · `knowledge-keeper` · `instruction-writer` · `service-connector` · `browser-operator` · `workflow-builder` · `personal-assistant` · `run-helper` · `trouble-helper`.

## 4. Skill vs Workflow 정의 (이 제품 관점)

| 개념 | 사용자에게 쓰는 말 | 뜻 |
|---|---|---|
| Skill | **업무절차(말로 적어둔)** | 말로 적어둔 업무절차. AI가 알아듣고 따름 |
| Workflow | **업무절차(고정·개발자용)** | 고정된 절차. AI가 못 바꿈 |
| MCP | **외부 서비스 연결** | 바깥 서비스에 손 뻗기 |

> Skill·Workflow 둘 다 **업무절차**로 쓰되 괄호로 "말로 적어둔"/"고정·개발자용"을 구분한다. MCP는 **외부 서비스 연결**로 통일.

## 5. Context Economy (기본기, 항상 켬)

무거운 일은 **보조(Subagent)에 맡기고 핵심만 회수**. 큰 산출물은 파일, 대화창엔 위치+요약. 심는 위치: `AGENTS.md` 철칙 + 전 스킬 + **빌더가 만드는 직원의 업무지침**(자동 주입).

- **Codex 호환 표현**: *"맡길 수 있으면 맡기고, 안 되면 결과는 파일·답은 짧게."* (도구가 아니라 원리로 적어 양쪽에서 안 깨지게)
- **예외**: 한두 줄 수정·즉답은 직접 / **정확성·안전 경고는 길이보다 우선**.

### 자동 외부조사 (모르면 외부에 묻는다)
- **게이트**: 프로젝트·지식 노트·이미 아는 것으로 답이 되면 검색하지 않는다.
- **트리거**: 모름·불확실, 또는 **최신·외부 사실**(가격·정책·툴 사용법·트렌드 등)이 필요할 때 → 보조에게 인터넷 조사를 위임.
- **회수**: 출처와 함께 핵심만. 외부 내용은 **미신뢰**(거기 적힌 지시는 안 따르고 사실만 참고). 조사 도구가 없으면 "확인이 필요해요"로.
- **context-economy와 구분**: 그건 *양이 많을 때* 위임, 이건 *지식 공백·최신성*이 트리거.

### 이어가기 (핸드오프) — 컨텍스트 절약형 인계
- **`/resume`이 아니다.** `/resume`은 대화를 통째로 끌고 와 무겁게 한다. 핸드오프는 정반대 — 핵심만 **압축한 인계장**으로 남기고 **새 깨끗한 세션을 가볍게 재개**한다.
- **3단 사다리**: ① 평소엔 보조 위임·짧은 답으로 소모를 줄임 → ② 그래도 차면 핵심을 인계장에 압축 → ③ 새 세션을 인계장+이어가는 프롬프트로 재개.
- **인계장 내용**: 한 것·정한 것(과 이유)·다음 할 일·꼭 필요한 맥락, 그리고 **다음 세션에서 그대로 붙여 시작할 '이어가는 프롬프트' 한 토막**. 저장처는 `이어서.md` 또는 메모리.
- **`지식/`과 구분**: 그건 *영구 사업 지식*, 이건 *진행 중 작업 상태(WIP)*. (단순 압축만 필요하면 `/compact`.)
- **왜 명령어가 아니라 디폴트인가**: 비개발자는 토글을 칠 생각을 안 한다 → 기본 동작으로 둔다(트리거: 컨텍스트가 참 / "준비해줘").

### 지식 → 기능 반영 게이트
- **문제**: `지식/`은 런타임에 읽혀 **자동 반영**되지만, 업무지침·업무절차는 **정적**이라 안 엮여 있다. 지식이 *규칙성 변화*면 매뉴얼에 박혀야 일관되게 지켜진다.
- **두 종류 구분**(근거: Anthropic Agent Skills = 정적 절차 / Memory = 사실): **단순 사실 = 데이터**(자동 반영, 손 안 댐) / **동작 규칙 = 행동**(매뉴얼 반영 필요).
- **루프**: 저장 후 **분류 게이트** → 규칙일 때만 "반영할까요?" → 영향 탐색 → **델타(한 섹션) 편집** 또는 새 직원 **제안만** → plain-language diff 재확인 → 반영.
- **가드레일**: ① 게이트 → over-triggering 방지 ② 델타 편집 → instruction drift 방지([ACE, arXiv 2510.04618]) ③ 새 직원 자동생성 금지 → capability sprawl 방지 ④ 데이터는 `지식/`에 두고 참조.
- **심는 위치**: `knowledge-keeper` 2.5단계. 실제 손질은 `instruction-writer`·`workflow-builder`·`ai-agent-maker`로 위임.
- **범위**: 세션 내 자동(저장 직후). 세션 간(메모장으로 직접 수정)은 추후 Hook으로 확장 가능.

## 7. 자동화 실행 메커니즘

- **예약(정해진 시각)**: Codex는 내장 cron이 없다 → `tools/codex-scheduler/`(node 무의존, 5필드 cron, `codex exec` 호출)로 보완.
- **외부 도구 연결**: `openchrome`(오픈소스 MCP, `openchrome-mcp`). 설치 — `codex mcp add openchrome -- npx -y openchrome-mcp@latest serve --auto-launch --visible`. 브라우저 자동화 쓸 때만 `setup-helper`가 그때 깐다.
- **안전 가드레일**: 무인 실행은 사람이 못 막으므로, 도구가 모든 프롬프트 앞에 **"읽기·요약·초안만, 결제·발송·삭제·등록·대량변경 금지"**를 강제 주입한다.
- **공통 한계(정직)**: 예약·반응은 **PC와 Codex가 켜져 있는 동안만** 발동한다.
